

### 成就感最大的事情

```
就是在大学期间参加的一个竞赛，也就是简历中的weTeam校园这个项目。当时是发现了一个比较好的点子，校园中缺少一个信息活动管理平台，去汇集一些校内外的活动信息，方便查询，并且提供一个组队的途径去帮助学生组队。 
但是没找到人一起去完成，我就只能自己一个人完成，而比赛的截止日期也很短，不到一个月时间，此外我对开发这个系统要用到各种技术还不熟，所以压力很大，也不知道能否如期完成。
所以之后就是处于一种边学习边开发的状态，白天上课，课余时间就开发系统，经过努力，还是在比赛前完成了系统，并且上线使用。最后，由于项目的点子很好，实现的功能也比较完善，评委老师给了很高的评价，并且帮助我在学院里面推广使用，最后拿到了比赛的一等奖

而我也通过自己的平台，找到了一些志同道合的队友，一起对系统进行完善改进，并一起参加之后的中国高校计算机大赛全国，并获得了西北赛区一等奖和全国三等奖的成绩。
这就是我成就感最大的事情。
```

### 挫败感最大的事情

```
挫败感最大的事情就是当时在开发中国航发商发这个项目时，出现了一些意料之外的事情，在快要验收的时候，客户修改了一些关键性的需求，这也直接影响到了我们的工作进度安排，面对可能不能按期完成任务，产生了一些挫败感。
但是经过一小段时间冷静后，意识到抱怨和沮丧是解决不了问题的，于是重新和客户反复确认了需求，将任务重新拆解分发下去，在开发的过程中也不断和客户去沟通，确保能够满足他们需求，最终还是如期交付项目
虽然这是一个非常困难的时期，但也从中学到了很多宝贵的经验，学会了如何去适应突发事件，如何更有效的去和客户沟通，以及在压力下保持冷静和专注。这次经历也使我更加意识到，即使在遇到挫败时，我们仍然有机会学习和成长。”
```

### 你认为在这个项目中哪些做的对，哪些做的不够好？

```
做的好的地方就是，
- 我会对这个项目需要开发的功能列一个表，列这个表就可以让我知道整体的进度，知道大致的规划。然后做完一个模块之后我会去对这个代码进行一个简单的回顾，做一些测试，看一下是否有问题，顺便清理一下一些无用的代码，加一些注释，特别是这个注释，我觉得很有必要，一方面是为了让别人能看懂，另一方面是为了能够让自己不会忘记。
- 对项目中用到的一些技术会给它记录沉淀下来。

做的不足的地方，就是项目前期时，在团队协作沟通方面，还有很大的提升空间。
- 比如有一些通用的方法，其他人可能也用到了，抽取出来之后没有很好的沟通，导致一些重复性的工作。
- 模块边界的地方一开始没有定好，导致各自开发完之后还需要去调整。
```

### 如果下次再有类似项目准备怎么做？

```
首先我可以通过现有的项目，能很熟悉整个开发流程，并且一些相同的业务就可以借鉴：

首先去充分理解和沟通每一个需求，保证需求是准确的
在需求的基础上，系统设计方面多下些功夫，因为一个好的系统架构才能够保证一个系统的各方面的性能

在开发时，踩过的一些坑避免再犯。
- 比如一些通用的方法，我们可以在开发写就商定写在一个特定的地方，并通过知识库来记录下来
- 在项目开发之前，更加注重系统的设计，对于一些模块的边界，一些协议，可以事先定下来，记录在知识库上，避免以后不必要的矛盾。
- 还有一些好的问题解决办法，大家可以去共享，去学习，在项目开发阶段，多去相互间沟通交流


一些优点也可以延续：
- 比如写完一个模块，去对这个代码进行回顾，做充分的测试，不要等到最后出问题后再去排查。
- 定期的回顾清理代码，加一些必要的注释，这样能帮助自己和他人快速理解，减少不必要的时间。
- 用到一些比较好的设计方法，设计模式，可以在其他项目中继续使用，再使用中不断完善
```

### 遇到困难如何解决，有什么收获

```
遇到困难之后难免会有压力，重要的是这种压力的情况下去调整自己的心态。
我在遇到困难时，通常也会有烦躁的时候，我会通过一小段时间的冷静、冥想给自己去放松一下。
然后回过神来去寻找解决的办法，可以通过一些开源的项目、优秀的博客或者向身边的朋友请教，
解决问题后会去回顾一下问题，去问自己一开始怎么没想到解决的办法。然后会把遇到的问题记录一下，避免以后再犯错误。
```

### 优点

```
对于一件事情，我可能会去准备一会再去做，把问题考虑的比较全。这样会让我对这件事有一个比较好的把握，能够知道这件事情大致的规划，心里也比较有底，之后开展起来也比较游刃有余。

但这样也会存在弊端，就是有一些考虑的东西可能在实际过程中没发生，是自己想多了，也就相当于浪费了一些时间
```

### 缺点

```
协作沟通方面还有待加强，因为在和他人一起开发项目的时候，
如果是同级的关系会比较棘手，大家都是一个年级的，每个人都有自己的想法，我也不能去指挥他们，所以有的时候沟通会比较少。这种问题目前也找到了一些方法去改进，我的做法就是，涉及到一些团队协作的时候，边界的地方，大家一起商定好，然后都遵照这个协议去开发，出了问题自己背锅。
```
